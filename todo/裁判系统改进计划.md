## 二、改进项

### P1: 上下文传递不完整（高优先级）

**问题**：裁判只收到原始任务 + 最近一次 assistant 回复，缺失中间对话轮次、工具调用结果、文件变更历史。

**当前代码** — `autoJudge.ts:74-89`：
```typescript
const originalTask = extractOriginalTask(fullMessages)
const lastAssistantText = assistantMessages
  .filter(m => m.type === 'assistant')
  .map(m => extractTextContent(m.message.content, '\n'))
  .join('\n\n')
```

**改进方案**：将完整对话历史传递给裁判

```typescript
// 方案 A：传递完整对话作为 context
const conversationContext = fullMessages
  .filter(m => !m.isMeta)
  .map(m => {
    const role = m.type === 'user' ? 'User' : 'Assistant'
    const text = extractTextContent(m.content, '\n')
    return `=== ${role} ===\n${text}`
  })
  .join('\n\n')

// 方案 B（推荐）：传递结构化摘要
// - 原始任务
// - 每轮 assistant 回复摘要
// - 工具调用列表（名称 + 关键参数）
// - 之前的裁判反馈（如果有）
```

**具体改动**：

1. 修改 `runAutoJudge()` 的 prompt 构建逻辑，包含：
   - 完整对话历史（或结构化摘要）
   - 文件变更列表（从 tool_use 中提取）
   - 之前裁判的 verdict 和 conciseIssues（多轮迭代场景）

2. 新增 `extractConversationSummary()` 函数：
   ```typescript
   function extractConversationSummary(messages: Message[]): string {
     const turns: string[] = []
     let turnNum = 0
     for (const m of messages) {
       if (m.type === 'assistant') {
         turnNum++
         const text = extractTextContent(m.message.content, '\n')
         const toolUses = m.message.content
           .filter((b: any) => b.type === 'tool_use')
           .map((b: any) => `${b.name}(${JSON.stringify(b.input).slice(0, 100)})`)
         turns.push(
           `### Turn ${turnNum}\n` +
           `Response: ${text.slice(0, 500)}\n` +
           `Tools: ${toolUses.join(', ') || 'none'}`
         )
       }
     }
     return turns.join('\n\n')
   }
   ```

**预期效果**：裁判能理解多轮迭代上下文，准确判断修复是否真正解决了之前的问题。

### P3: 添加循环上限（高优先级）

**问题**：`query.ts:1403` 的 `continue` 无计数限制，可能导致无限循环、token 耗尽、费用失控。

**当前代码** — `query.ts:1401-1403`：
```typescript
state = {
  // ...
  transition: { reason: 'auto_judge_retry' },
}
continue  // 无上限
```

**改进方案**：

1. 在 state 中添加裁判重试计数器：
   ```typescript
   // bootstrap/state.ts — State 类型新增字段
   autoJudgeRetryCount: number
   ```

2. 在 query.ts 中检查上限：
   ```typescript
   const MAX_AUTO_JUDGE_RETRIES = 5  // 可配置

   if (autoJudgeResult.verdict !== 'PASS') {
     if (autoJudgeRetryCount >= MAX_AUTO_JUDGE_RETRIES) {
       logEvent('tengu_auto_judge_max_retries_exceeded', {
         verdict: autoJudgeResult.verdict,
         retryCount: autoJudgeRetryCount,
         queryChainId: queryChainIdForAnalytics,
       })
       // 达到上限，正常结束但标记为未通过验证
       return { reason: 'completed', autoJudgeFailed: true }
     }
     // ... 现有的 continue 逻辑
   }
   ```

3. 在 UI 层显示警告：
   - 当重试次数 ≥ 3 时，显示系统提示："裁判已重试 N 次，模型仍在修复中..."
   - 达到上限时，显示："裁判在 5 次重试后仍未通过，请人工检查。"

**配置建议**：
- 默认上限：5 次
- 可通过环境变量覆盖：`CLAUDE_MAX_AUTO_JUDGE_RETRIES`
- 可通过设置文件配置：`.claude/settings.json` → `"max_auto_judge_retries": 3`

### P4: 添加超时机制（中优先级）

**问题**：单次裁判执行和整体循环均无超时保护。

**改进方案**：

1. 单次裁判超时：
   ```typescript
   // autoJudge.ts — 在 runAutoJudge 中添加超时
   const JUDGE_TIMEOUT_MS = 120_000  // 2 分钟

   const timeoutPromise = new Promise<never>((_, reject) => {
     setTimeout(() => reject(new Error('Judge timed out')), JUDGE_TIMEOUT_MS)
   })

   // 使用 Promise.race 包装 agent 执行
   const agentResult = await Promise.race([
     runAgent(...),
     timeoutPromise,
   ])
   ```

2. 整体循环超时：
   - 在 state 中记录进入裁判循环的时间
   - 每次循环检查总耗时
   - 超过阈值（如 10 分钟）强制退出

### P5: VERDICT 解析失败处理优化（低优先级）

**问题**：`autoJudge.ts:112` 解析失败时默认返回 `'PARTIAL'`，可能导致误触发修复循环。

**当前代码**：
```typescript
const verdict = parseVerdict(report) ?? 'PARTIAL'
```

**改进方案**：

```typescript
const verdict = parseVerdict(report)
if (verdict === null) {
  // 裁判未输出有效 VERDICT，记录警告但不触发修复循环
  logForDebugging('Judge did not output a valid VERDICT line')
  logEvent('tengu_auto_judge_no_verdict', {
    reportPreview: report.slice(0, 200),
    queryChainId: queryChainIdForAnalytics,
  })
  // 默认 PASS：宁可放过，不可错杀
  return { verdict: 'PASS', logFilePath, conciseIssues: '' }
}
```

**理由**：裁判本身是 LLM，可能因各种原因（网络错误、prompt 被截断等）未输出 VERDICT 行。此时默认 PASS 比默认 PARTIAL 更安全，避免无效循环。

---

### P6: 共享 abortController 问题（低优先级）

**问题**：`runAgent.ts:524-528` 中，同步 agent（`isAsync: false`）共享父级的 abortController，主线程取消会同时取消裁判。

**当前代码**：
```typescript
const agentAbortController = override?.abortController
  ? override.abortController
  : isAsync
    ? new AbortController()
    : toolUseContext.abortController  // 共享
```

**改进方案**：

裁判应该始终拥有独立的 abortController，不受主线程取消影响：

```typescript
// autoJudge.ts 中调用 runAgent 时
const judgeAbortController = new AbortController()

for await (const message of runAgent({
  agentDefinition: VERIFICATION_AGENT,
  promptMessages: [createUserMessage({ content: verificationPrompt })],
  toolUseContext,
  canUseTool,
  isAsync: false,
  querySource,
  availableTools: toolUseContext.options.tools,
  override: { abortController: judgeAbortController },
})) {
  // ...
}
```

## 四、额外建议

### 4.1 裁判结果可视化

在终端中显示裁判进度和结果：

```
🔍 Running verification...
  ✓ Build passed
  ✓ Test suite: 42/42 passed
  ✗ POST /api/users - duplicate email not rejected
  ⚠ Boundary test: empty string → 500 (expected 400)

❌ VERDICT: FAIL — 2 issues found
  See .claude/logs/judge/2026-04-04T12-30-45-turn3-fail.md
```

### 4.2 裁判模型可配置

当前裁判继承主模型（`model: 'inherit'`），建议允许配置独立的裁判模型：

```json
{
  "auto_judge": {
    "enabled": true,
    "model": "claude-sonnet-4-20250514",
    "max_retries": 5
  }
}
```

### 4.3 裁判缓存优化

裁判的 prompt 包含大量重复内容（原始任务、对话历史），可以利用 prompt cache 减少重复 token 消耗：

- 原始任务 + 对话历史使用 `cache_control: { type: "ephemeral" }`
- 每次只发送新增的 assistant 回复

### 4.4 裁判对抗性增强

在 verification prompt 中添加：
- 之前裁判反馈的历史记录
- 主模型针对之前反馈的修复说明
- 要求裁判重点验证修复点是否真正生效
