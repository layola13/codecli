# 自动裁判系统实现

## 一、需求

用户期望实现一个自动裁判系统：

1. 每次大模型完成一轮工作后，**自动**生成一个裁判 agent 审核
2. 裁判检查用户需求是否真正被满足，给出判断过程和最终判定（PASS/FAIL/PARTIAL）
3. 如果判定不及格（FAIL/PARTIAL），大模型**继续执行修复，不停止**
4. 循环执行，直到裁判判定 PASS

用户发现已有系统"好像无效"——裁判没有真正发挥作用。

## 二、已有系统分析

### 2.1 已有组件

| 组件 | 文件 | 状态 |
|------|------|------|
| Verification Agent | `src/tools/AgentTool/built-in/verificationAgent.ts` | 存在且设计精良 |
| Judge Mode 状态 | `src/bootstrap/state.ts` | 存在，默认关闭 |

| Agent 执行机制 | `src/tools/AgentTool/runAgent.ts` | 存在，可复用 |

### 2.2 Verification Agent 设计亮点

- **对抗性策略**：要求并发测试、边界值、幂等性、孤立操作等探测
- **命令执行要求**：每个检查必须有 `Command run` + `Output observed` + `Result`，禁止仅阅读代码
- **反模式识别**：明确列出 "verification avoidance" 和 "seduced by the first 80%" 两种 LLM 常见失败模式
- **输出格式**：严格要求以 `VERDICT: PASS` / `VERDICT: FAIL` / `VERDICT: PARTIAL` 结尾

### 2.3 为什么无效

| 问题 | 原因 |
|------|------|
| 默认关闭 | `judgeModeOptIn: false`，需要手动 `/judge on` |
| 被动触发 | 启用后仅注册为"可用 agent"，依赖主模型主动调用 |
| 无强制机制 | 系统提示只是建议 "Before reporting a task complete, run the verification agent" |
| 无 FAIL 循环 | 即使裁判打出 FAIL，主模型也可以直接停止 |

**根本原因**：缺少代码层面的自动触发和结果强制执行机制。

## 三、实现方案

### 3.1 用户决策

| 决策点 | 选择 |
|--------|------|
| 触发时机 | 每次主模型响应结束后自动触发（不判断是否"完成"） |
| 评分方式 | 保持 PASS/FAIL/PARTIAL 三档，不引入数字评分 |
| 循环上限 | 不设上限，FAIL 后持续循环直到 PASS |

### 3.2 核心思路

在 `queryLoop()` 的主循环中，当模型完成一轮响应（无更多 tool_use）、stop hooks 执行完毕后，插入自动裁判拦截点：

```
主模型完成一轮响应（无更多 tool_use）
        ↓
stop hooks 执行完毕
        ↓
token budget 检查
        ↓
  ★ runAutoJudge ★
  启动 verification agent（后台）
        ↓
    解析 VERDICT
    ↙           ↘
  PASS      FAIL/PARTIAL
    ↓            ↓
正常结束    verdictMsg 注入对话
            state = {..., continue}
            主模型继续修复 → 再次触发裁判
            循环直到 PASS
```

### 3.3 代码变更

#### 新增文件

**`src/judge/parseVerdict.ts`** — VERDICT 解析器

```typescript
export type Verdict = 'PASS' | 'FAIL' | 'PARTIAL'
const VERDICT_RE = /VERDICT:\s*(PASS|FAIL|PARTIAL)/
export function parseVerdict(text: string): Verdict | null {
  const m = text.match(VERDICT_RE)
  return m ? (m[1] as Verdict) : null
}
```

**`src/judge/autoJudge.ts`** — 自动裁判核心逻辑

- `runAutoJudge()` — async generator，调用 `runAgent(VERIFICATION_AGENT)` 启动裁判
  - 将主模型最近的 assistant 消息作为验证输入
  - 流式 yield 裁判执行进度（供 UI 展示）
  - 从最终输出中解析 `VERDICT: PASS/FAIL/PARTIAL`
  - 返回 `{ verdict, report }`
- `createVerdictFeedbackMessage()` — 将裁判报告包装为用户消息，注入对话

#### query.ts 关键代码

```typescript
// Auto-judge: run verification agent after each turn completion.
// If the verdict is FAIL or PARTIAL, inject the report and continue
// the loop so the main model can fix the issues.
const autoJudgeResult = yield* runAutoJudge({
  assistantMessages,
  toolUseContext,
  canUseTool,
  querySource,
})

if (autoJudgeResult.verdict !== 'PASS') {
  const verdictMsg = createVerdictFeedbackMessage(
    autoJudgeResult.verdict,
    autoJudgeResult.report,
  )
  logForDebugging(`Auto-judge verdict: ${autoJudgeResult.verdict}`)
  logEvent('tengu_auto_judge_verdict', {
    verdict: autoJudgeResult.verdict,
    queryChainId: queryChainIdForAnalytics,
    queryDepth: queryTracking.depth,
  })
  state = {
    messages: [...messagesForQuery, ...assistantMessages, verdictMsg],
    toolUseContext,
    autoCompactTracking: tracking,
    maxOutputTokensRecoveryCount: 0,
    hasAttemptedReactiveCompact: false,
    maxOutputTokensOverride: undefined,
    pendingToolUseSummary: undefined,
    stopHookActive: undefined,
    turnCount,
    transition: { reason: 'auto_judge_retry' },
  }
  continue
}

logEvent('tengu_auto_judge_passed', {
  queryChainId: queryChainIdForAnalytics,
  queryDepth: queryTracking.depth,
})

return { reason: 'completed' }
```

## 七、落地复核（2026-04-04）

这份文档在复核时**不算完全落地**，当时代码与文档至少有以下不一致：

1. 文档写了默认开启（`judgeModeOptIn: true`），实际代码仍是 `false`
2. `/judge` 只支持空参切换和 `on/off`，没有显式的 `/judge toggle`
3. 启动画面还没有按需求显示 `JUDGE on` / `JUDGE off`
4. `· Noodling…` 这类主 spinner 提示没有随 judge 状态切成绿色
5. 自动裁判跑的是 `VERIFICATION_AGENT`，但主循环没有明确避开 subagent/verification 自身，存在递归验裁判风险
6. judge 汇总上下文时没有把“当前这一轮”的 assistant 消息并入完整会话摘要，当前轮工具调用和改动感知不完整

本次补齐后，现状如下：

1. `/judge toggle` 已支持，`/judge on|off|toggle` 都可用
2. 启动画面会始终显示 `JUDGE on` 或 `JUDGE off`
3. judge 开启时，主 spinner 文案会切到绿色；关闭时恢复原色
4. 自动裁判只在主线程 `repl_main_thread*` 上运行，避免对子 agent，尤其是 verification agent 自己递归触发
5. judge 读取的上下文已包含当前轮 assistant 消息，因此当前轮响应和工具改动会被纳入核查
6. judge 日志会写到项目根目录下的 `.claude/logs/judge/`

仍保留的一点差异：

1. 默认值目前仍是 `judgeModeOptIn: false`，没有改成文档里写的默认开启
